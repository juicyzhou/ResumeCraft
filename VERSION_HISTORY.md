# ResumeCraft 版本演进记录

本文档根据仓库 Git 提交历史整理，覆盖从首个提交到当前版本的全部 17 个版本节点。日期均为北京时间（UTC+8）。

> 版本说明：仓库尚未使用语义化 Git Tag，因此本文以提交顺序编号为“版本 01—17”，并保留提交短哈希，便于定位、比较和回滚。
>
> ATS 说明：项目通过单栏阅读顺序、低风险排版和导出前检查降低解析风险，但不同招聘系统的解析规则并不完全公开或一致，因此不应将任何网页模板表述为对所有 ATS 的绝对 100% 保证。

## 版本 01 — 初始简历编辑器

- 提交：`9bfe440`（Build Jianxu resume editor）
- 作者：Codex Sites
- 日期：2026-07-15 23:43

主要改进：

- 建立完整的网页版简历编辑工作台，采用“章节导航 + 表单编辑 + 实时预览”三栏结构。
- 首批提供“清简、新章、松岚”3 套模板，覆盖通用、现代和克制型视觉风格。
- 支持在表单和预览区域直接修改内容，实现基础所见即所得体验。
- 建立基本信息、个人简介、工作经历、项目经历、教育经历、专业技能 6 个内置章节。
- 使用浏览器 `localStorage` 自动保存草稿，并展示保存状态。
- 支持预览缩放、打印预览、导出 PDF、移动端编辑/预览切换及恢复示例。
- 初始化 Next.js、Vite/Sites、数据库示例、测试和基础工程配置。

用户价值：完成从内容填写、视觉预览到 PDF 输出的首个可用闭环。

## 版本 17 — 数据可靠性、章节库、岗位示例与 ATS 验证

- 提交：`3e5d9d7`（Add resume backup history sections and ATS checks）
- 作者：juicyzhou
- 日期：2026-07-19 00:09

主要改进：

### 本地数据备份

- 新增完整 JSON 备份导出与导入，覆盖内容、模板、技能模式、ATS 设置、章节顺序和示例类型。
- 界面明确提示数据只保存在当前浏览器，重要修改可下载到本机保存。
- 本地存储升级到 `jianxu-resume-v4`，并继续兼容 v1—v3 历史数据。

### 撤销与恢复

- 新增最多约 50 步的撤销/重做历史栈。

### 结构化章节库

- 自定义章节从单纯文本扩展为文本、列表、时间轴三种结构。
- 新增校园经历、荣誉奖项、证书认证、论文发表、专利成果、开源贡献、作品集、语言能力等预设章节。
- 每类结构化记录支持标题、补充信息、时间和成果正文，并兼容模板、排序、分页和 ATS 文本导出。

### 岗位示例

- 新增校招开发工程师、资深工程师、AI 算法工程师、产品经理 4 套针对性示例。
- 保留原有 AI 工程精简版和完整多页版，共形成 6 类可切换示例。
- 示例内容覆盖实习、架构治理、模型训练评测、论文、开源、产品指标和作品集等不同评价重点。

### ATS 导出验证

- 新增 ATS 规范化文本生成器，按照用户设置的章节顺序线性输出全部内容。
- 导出前检查姓名与求职方向、联系方式、邮箱格式、正文完整度、特殊符号、阅读顺序、页数和重复章节。
- 提供兼容性检查分、通过/警告/失败明细、规范化文本预览和文本下载。
- 存在必填失败项时阻止继续导出 ATS PDF；同时明确结果用于降低解析风险，不代表录取结论。
- 新增自动化测试，验证章节顺序、结构化章节不重复以及缺失身份/联系方式时能正确报错。

### 其他工程改进

- 增加社交分享图和页面元信息优化。
- 更新依赖与测试命令，使 ATS 校验进入项目测试流程。

用户价值：编辑器从单纯的排版工具升级为包含数据安全、内容结构、岗位参考和导出质量控制的完整简历工作台。

## 阶段性演进总结

1. **基础编辑（版本 01—02）**：完成三栏编辑、所见即所得、本地保存、PDF 导出和工程岗位内容。
2. **模板体系（版本 03—07）**：从单纯换色扩展到单栏、双栏、时间轴、Markdown、学术、高密度和横章等 15 套模板，并持续打磨配色。
3. **ATS 与发布（版本 04—10）**：形成严格模式、5 套 ATS 主题、双模式导出和 GitHub Pages 静态发布能力。
4. **内容结构（版本 11—12、15—17）**：工作、项目、教育和自定义章节均实现动态化，并扩展成专业章节库和岗位示例库。
5. **真实分页（版本 13—16）**：从固定分页、连续分页线演进到与 PDF 一致的 A4 页壳、智能切点和防孤立标题。
6. **可靠性与验证（版本 17）**：补齐本地备份、撤销重做、数据迁移、ATS 预检和自动化测试。

## 维护建议

- 后续正式发布时建议添加 Git Tag（如 `v1.0.0`）和 GitHub Release，使用户版本与提交版本对应。
- 每次功能提交继续记录“用户可感知变化、数据兼容性、导出影响和回滚点”，避免更新日志只描述技术实现。
- ATS 相关文案应持续使用“兼容性检查、降低解析风险”，避免承诺无法跨厂商验证的绝对兼容。
